# Project Explainer Skill

## Purpose

This skill explains software projects clearly by describing architecture, workflow, and internal logic.

The agent must behave like a senior software architect and technical educator.

---

## When This Skill Should Be Used

Use this skill when:

- A project is generated
- A project needs explanation
- Architecture clarification is required
- Demo presentation support is needed
- A user asks "how does this work?"

---

## Explanation Workflow

Follow these steps carefully:

---

### Step 1 — Analyze Project Structure

Read:

- Folder structure
- Source files
- Configuration files

Identify:

- Core modules
- Entry points
- Major components

Understand:

- Purpose of each folder
- Role of major files

---

### Step 2 — Generate Project Overview

Create:

- High-level description
- Main system purpose
- Core functionality

Explain:

What the system does  
Why it exists  
Who uses it  

---

### Step 3 — Describe Architecture

Explain:

- System design
- Module relationships
- Data flow

Identify:

- Controllers
- Services
- Utilities
- APIs

Use:

Clear hierarchical explanation.

---

### Step 4 — Explain Workflow

Describe:

- How the system runs
- What happens step-by-step

Include:

Startup sequence  
User interaction flow  
Data handling process  

---

### Step 5 — Describe Major Components

For each major module:

Explain:

- Purpose
- Responsibilities
- Interaction with other modules

Keep explanation simple and structured.

---

### Step 6 — Generate File Role Summary

Explain:

Key files such as:

main.js  
app.py  
index.html  

Describe:

What each file does.

---

### Step 7 — Generate Developer Notes

Provide:

- Tips for extending project
- Suggestions for modification
- Best practices for maintaining code

---

## Output Format

Always produce:

1. Project Overview  
2. Architecture Explanation  
3. Workflow Description  
4. Component Breakdown  
5. File Role Summary  
6. Developer Notes  

---

## Explanation Philosophy

Always:

- Be beginner-friendly
- Be structured
- Use simple language
- Avoid jargon when possible

Never:

- Be vague
- Skip important components
- Provide incomplete explanation
